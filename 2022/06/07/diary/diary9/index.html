Diary

额，7 号又忘了写日记。 今天补，哈，今天补...

这几天感觉人越来越皮了，咳，得改改。喏，那今天就来点严肃的。

那个，写的有点枯燥。如果觉得没意思，直接看最后几句话就行。

北院，下午二点。

昨天收到通知，老师说今天开会。本来以为开会的主题是我们的实习项目呢，结果下午开会的时候，研究生、另一组及本组三批人坐在了一起，当时人还有点懵，直到老师说今天要讲建模，自己才反应了过来。

怎么说呢，这次开会主要讲了两个内容。一个是介绍常用的建模软件，一个是初步了解建模软件 UDEC。大部分时间主要用在了第二个内容上，这个内容主要围绕三方面来讲述：

  1. 学习方法
  2. 建模流程
  3. 示例学习

额，这具体内容就不说了...

说些有意思的地方，老师在一开始写 udec 代码的时候，他期间更换了 4 种文本编辑器，即 Visual Studio Code、Gvim、Notepad（记事本） 以及 Typora。这不是重点，重点是老师在这几种编辑器间来回切换，看的我头大。怎么说呢，对于文本编辑，他没有正确的认知，只会合乎心意地选择当前好用、自己熟悉的编辑器和编辑模式。

为什么这么说，我来举个例子。一般来说，上面除了 Gvim 这个软件外的其他几个软件，都有一个特点，即，他们的编辑模式相同（插入模式）。而老师在实际使用 Visual Studio Code 时，他觉得有些操作不太好用，于是启用了 neovim 插件，切换到 Vim 编辑模式，但是呢，他又对 Vim 编辑模式不熟，然后又停用了 neovim 插件。我...

老师在带我们跟 udec 手册的示例学习的时候，遇到过这样一个问题，即，手册中的代码复制过来后会被拆行，就像下面示例：

对于这个问题，老师是怎么做的呢？他一开始在 Visual Studio Code 意识到这样不好处理，所以就立马用 Gvim 打开文件，快速将后三行内容与第一行内容合并，我看了下，他总共重复了三次操作。

我当时的第一反应是这是啥操作，我经常用 vim(neovim) 咋没用过，所以我有点好奇。我想了想，要是我遇到这种问题，我会怎么做？下意识的反应：vipJ。但实际情况是，我只对汉字用过行合并，一般英文不会遇到这种需求，所以我不知道 vipJ 合并英文行，英文间会不会自动加空格。我回到宿舍立马试了试，答案是可以。就像下面动图一样：

这就完了吗？并没有。老师那操作，我一直想不明白那个功能是单独将下一行的内容合并到上一行，而且不用进行选择。最后想了想会不会是 J 本身的功能，试了下，猜的没错，所以老师在 Gvim 上做的操作是 JJJ。也许会有人想，到底谁的速度快呢，在面对几十上百行这样问题时，vipJ 合并只要一瞬间，老师那方法要按几十上百次 J。

至于到底为什么可以这样做，请移步 Vim。

为什么要说这个事呢？老师那 Gvim 只用了一次，后来一直出现这种问题是，他选择了在 Visual Studio Code 用最笨的方法，鼠标点到下一行行首，回退，插入空格。他本来想在 Visual Studio Code 中用之前的操作，所以他启用了neovim 插件，但是呢，出了问题，他立马切换编辑模式（停用 neovim 插件），然后不得不用那笨办法。

我想看到这，你应该看出他的问题了，对于不熟悉的编辑模式，只要遇到不会的问题，就立马切回常用的编辑模式。我曾经跟他聊过 vim(neovim)，他觉得 vim（neovim）那不是一般人能学会的，我当时内心是不大认同的，你如果喜欢，如果愿意花时间，遇到问题主动的寻求解决方案，那有什么难的？世界上几百万人都会的东西，你凭什么说它难呢？

我写这不是为了炫耀我会 vim(neovim)，也不是看不起老师。我会的只是些皮毛而已，老师在许多方面都比我强。

我只想说，如果你真正喜爱一个东西，那么，请你认真对待。
